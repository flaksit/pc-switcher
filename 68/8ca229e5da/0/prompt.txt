### CR-01 Fix Choice: Path Filtering vs. Re-ordering

The planner chose path filtering (exclude `.local/share/pc-switcher/` from divergence scope) and justified rejecting re-ordering on the grounds that the lock file is released in `_cleanup()` (the `finally` block at orchestrator.py:302–304) AFTER `_update_sync_history()`. This justification is directionally sound: even if baseline capture were moved to after line 284 (`_update_sync_history()`), the lock file is held by a persistent remote process that exits during cleanup, potentially modifying file state after the baseline. However, the planner's justification does not need to be airtight, because path filtering is independently justified and explicitly offered as a valid fix option in `01-VERIFICATION.md` ("OR have `_target_diverged_since` ignore paths matching `.local/share/pc-switcher/`..."). It is also architecturally cleaner: it stays entirely within `FolderSyncJob`, requires no orchestrator changes, and is robust to any future tool-state additions under that directory.

**Is the filter too broad?** No. `.local/share/pc-switcher/` is a tool-owned directory (`sync_history.HISTORY_DIR`, lock.py `get_lock_path()`). It contains only `sync-history.json`, `pc-switcher.lock`, and `logs/`. No user legitimately stores files there. The filter cannot mask real user divergence.
