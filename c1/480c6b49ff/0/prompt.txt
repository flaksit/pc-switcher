### CR-01 (BLOCKER, false negative → data loss)
`src/pcswitcher/jobs/folder_sync.py:381-397`. The tool-state exclusion uses unanchored substring matching (`any(token in line ...)` with token `"/.config/pc-switcher/"`). Because `btrfs find-new` paths are relative to the subvolume root, a real user file at *any* depth — e.g. `janfr/dotfiles/.config/pc-switcher/config.yaml` from a dotfile manager (chezmoi/yadm/stow), which the power-user target audience commonly runs — is silently classified `CLEAN`. The guard then lets `rsync --delete` overwrite genuinely-diverged target data. Tests only cover the depth-1 case.

### CR-02 (BLOCKER, false positive → breaks upgrade workflow, pushes to unsafe `--allow-divergence`)
The new WR-03 pre-transfer re-check (`folder_sync.py:577-586`) runs in Phase 9, *after* Phase 7 `install_on_target`. I verified via `install_on_target.py:70-75` and `install.sh:111-114,190-200` that an upgrade writes `~/.local/bin/pc-switcher`, `~/.local/bin/uv`, and `~/.local/share/uv/…` under the synced `@home` subvolume. The tool-state filter masks only the two `pc-switcher` state dirs, so `find-new` since the prior baseline reports these install artifacts as changes → `DIVERGED` → the sync aborts before rsync. This falsely blocks every sync that also upgrades the target (frequent during development where the source dev-version changes each run). It's a regression the old Phase-4-only check never exposed.

### WR-01
`lstrip("~/")` (folder_sync.py:381-382, config_sync.py:317) is character-set removal, not prefix removal — currently correct for the constants but a silent landmine feeding the security-relevant filter tokens. Recommend `removeprefix("~/")`.
