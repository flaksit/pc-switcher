### Key Findings

- **rsync-as-root wiring:** Local `sudo rsync -aAXHS --numeric-ids --delete` with remote `--rsync-path='sudo rsync'`. rsync spawns its own SSH transport independently from the asyncssh orchestration connection. Both source and target need `NOPASSWD: /usr/bin/rsync` in sudoers. The asyncssh connection is NOT reused for rsync's data channel â€” this is expected and correct per ADR-002's intent.

- **Divergence detection (D-08 resolved):** Use `btrfs subvolume show &lt;mount_point&gt;` to read the subvolume's "Generation:" field. Post-sync: record target's generation per folder in an extended `sync_history.json` (`target_generations[hostname][path]`). Pre-sync: compare current generation to stored value â€” any increase means writes happened after the last sync. This is O(1), reuses existing btrfs infrastructure, and requires no snapshot comparison.

- **rsync progress streaming (carriage-return problem):** `--info=progress2` uses `\r` not `\n`; the existing `LocalProcess.stdout()` iterator blocks on `\n` and will freeze the TUI during large transfers. Must use chunk-based reading (`proc.stdout.read(N)`) splitting on both `\r` and `\n`. `--out-format='%i %n%L'` provides per-file events on newlines (feeds FULL-level log).

- **`config-schema.yaml` additionalProperties guard:** The existing schema has `additionalProperties: false` at the top level. Adding `folder_sync:` to config.yaml will break validation until the schema is updated with a `folder_sync:` property AND `sync_jobs.properties` gains a `folder_sync: boolean` entry.

- **`--delete` is safe without `--delete-excluded`:** Machine-specific excluded files on the target (`.ssh/id_*`, Tailscale config) are protected from deletion by rsync's default behavior. Never pass `--delete-excluded`.
