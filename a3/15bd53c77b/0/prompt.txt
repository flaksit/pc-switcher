## Debug checkpoint reached

**Type:** human-verify — the agent applied and self-tested a fix but cannot exercise the actual reported workflow (a real p17 ↔ fleksi sync).

**Root cause:** `~/.config/Code/User/globalStorage/state.vscdb` (a single opaque SQLite file) was inside the synced `Code/User` tree with no exclude. It mixes two machine-bound things into one blob:
- **Trust list** (Symptom A) — `content.trust.model.key` holds the *entire* trusted-folders array as one JSON value. Confirmed on this machine: post-sync it contained exactly one entry — the folder just re-trusted — matching the report.
- **Extension auth** (Symptom B) — GitHub-authentication's session (and other extensions') live under `secret://...` keys as ciphertext (1230 bytes for `github.auth`). VS Code's own docs confirm the Linux decryption key is a password held in gnome-libsecret, generated independently per machine and never synced — so ciphertext copied from another machine is permanently undecryptable there, forcing re-login. This is VS Code's documented-by-design behavior; full-tree sync just exposes it.

**Fix applied** (diff shown above):
- `src/pcswitcher/home.filter` — excludes `state.vscdb*` (covers `.backup`, `-wal`, `-shm`, `-journal`) with an explanatory comment.
- `docs/configuration.md` — new subsection explaining both failure modes and the tradeoff (some harmless cross-machine state like terminal history also stops syncing).
- `tests/local_rsync/test_folder_sync_filters.py` — new `TestVSCodeStateVscdbExclude`.

**Self-verified:** full `pytest` suite (643 passed), `ruff check`/`format --check`, `basedpyright` all clean. I independently re-ran the filter test suite — 9/9 pass.
