<objective>
Give `folder_sync` an include-override capability (exclude a folder but keep select subfolders) by adopting native rsync filter syntax through two authoring surfaces: a per-folder central `merge` filter file and a tree-wide per-directory `dir-merge /.pcswitcher-filter`. This replaces the exclude-only `excludes: list[str]` config surface with a per-folder `filter_file` field, ships starter filter files via `init`, and documents the rsync filter model for users coming from `.gitignore`. Implements GitHub issue #166.

Purpose: users need to keep dev caches like `~/.cache/uv` while dropping the rest of `~/.cache` on a `--delete` mirror — impossible with exclude-only rules; native rsync filters (first-match-wins, `+`/`-`) express this directly.

Output: filter-rule code + schema, starter home.filter/root.filter, init wiring, unit tests, real-rsync acceptance tests, migrated e2e config, and updated ADR-016 + configuration.md.
</objective>

<critical_decisions>
Locked decisions from CONTEXT.md that constrain every task — do not revisit:

- filter_file is PER-FOLDER: an optional `filter_file: str | None` field on FolderEntry that REPLACES `excludes`; unset/empty ⇒ no central merge rule for that folder (runtime excludes + dir-merge still apply).
- Per-folder rsync filter rule order, GLOBAL-FIRST: (1) runtime-protection excludes (ADR-016, first, un-overridable), (2) `--filter='merge <expanded_abs_filter_file>'` when set, (3) `--filter='dir-merge /.pcswitcher-filter'` always.
- Missing filter_file (set but file absent on source) ⇒ ValidationError in validate(), source-side, alongside the existing `test -d` folder check; expand `~` and env vars BEFORE the check and BEFORE passing to rsync's merge.
- init ships starter home.filter + root.filter as package data alongside default-config.yaml; init writes them next to config.yaml, honoring --force.
- NEVER pass `-F` / `-FF` / `-C` / `--cvs-exclude`; dir-merge uses NO `e` modifier (filter files sync to target), NO `n` modifier (inheritance stays on), NO `--delete-excluded` (unchanged).

FLAGGED PLANNER DECISION (deviation from CONTEXT's leading-slash idiom, required for correctness): CONTEXT's ancestor-descent idiom is written with leading slashes (`+ /.cache/uv/***`), which anchor to the TRANSFER ROOT; but pc-switcher syncs `/home` and `/root` with the folder CONTENTS as the transfer root (`_build_rsync_cmd` sets src to `<path>/`), so each user's home sits ONE LEVEL BELOW the root (`janfr/.cache/uv`, not `/.cache/uv`), meaning a leading-slash `/.cache/...` would anchor to `/home/.cache` (a nonexistent path) and silently fail to keep `.cache/uv`, breaking acceptance #1 — therefore the shipped home.filter/root.filter use FLOATING (non-leading-slash) patterns exactly as today's `excludes` do (default-config.yaml line 124 comment confirms `.ssh/id_*` floats to match every user under /home), the leading-slash form is presented in the docs guide as the transfer-root-relative idiom with an explanation of anchoring, and the real-rsync acceptance test in Task 3 MUST build a `<root>/<user>/.cache/...` tree to prove the floating filter works at a /home-style transfer root.
