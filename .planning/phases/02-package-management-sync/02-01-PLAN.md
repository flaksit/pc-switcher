  <action>
Create ADR-020 titled "Declarative package convergence: manifest capture, item diff, replay via each ecosystem's own tooling", status Accepted, following the section structure ADR-001 prescribes.

Record these positions, each traceable to its CONTEXT decision id:

Context section: Phase 2 must replicate presence, version and provenance of packages across apt, snap and flatpak plus the repository/keyring/pin/remote state those installs depend on. Package data under `~/.var/app`, `~/snap/<app>/common` and dotfiles is already Phase 1 folder_sync territory.

Decision section, one subsection per position:
- Convergence model (D-01): the source captures a manifest; the target diffs its own state against it and converges using `apt`, `snap` and `flatpak` themselves. `/var/lib/dpkg`, `/var/lib/snapd` and the flatpak OSTree store are never rsynced — the package managers stay authoritative for their own state.
- Item model (D-02): every handled thing is an item with a stable identity, not just packages. Name the classes: apt package, apt source, apt key, apt pin, apt config file, snap, snap channel, flatpak ref, flatpak remote, unreproducible/manual install. All classes flow through one diff → decide → apply pipeline.
- Manifest content for apt (D-03): the manually-installed set from `apt-mark showmanual`, not the full dpkg selection set; apt resolves dependencies on the target.
- Version policy (D-04, D-05): versions float to whatever the target's repos currently offer; version mismatch is a reported diff class, never a forced downgrade. Deliberate pinning replicates because `/etc/apt/preferences.d` entries are items.
- Revision/scope convergence (D-06): snap manifest carries name + channel + revision, flatpak carries ref + origin + user/system scope. Convergence must never leave a snap held or otherwise blocked from auto-refresh; record `snap install --revision=N` / `snap refresh --revision=N` as the mechanism and `snap refresh --hold` as rejected.
- Three-way decision and direction (D-07): apply / skip once / skip always for every item class. "Apply" is direction-dependent — missing on target means install/add/enable, extra on target means remove/delete/disable, different on both means change the target to match the source. The review names the concrete action per item.
- Machine-local decision file (D-08, D-08a, D-09, D-10): one file per manager under `~/.config/pc-switcher/`, never synced, excluded from folder_sync and outside config_sync. An entry on machine M makes the item inert on M in both roles. The entry is written on the end of the connection that holds the item. Defaults ship as an example file only.
- Repository state as items (D-11, D-12, D-13): apt sources, keys, pins, apt config, flatpak remotes and snap channels are inventory items, not a file mirror; a `--delete` mirror of `/etc/apt` would wipe the target's own machine-specific sources. A repo's signing key travels bundled with its repo item, byte-for-byte; keys are never re-fetched from vendors. Legacy `/etc/apt/trusted.gpg.d` keys form a separate global-trust item set.
- Job split (D-15, D-16, D-17): three jobs `apt_sync`, `snap_sync`, `flatpak_sync` over one shared core extracted while building; package jobs run before folder_sync so apps are provisioned before their data lands.
- Unreproducible items and snippets (D-18 through D-23): detection of no-repo-candidate apt packages and unowned installs under `/usr/local` and `/opt`; an install snippet is an opaque text blob replayed non-interactively through the existing executor with the exit code deciding success; snippets live in the shared synced config and cover bare .debs and manual installs only.
- Review, failure and dry-run (D-24 through D-28): one batched review grouped by manager and action before any change; conflicts and version mismatches are diff classes inside that review, not a second reporting mechanism; non-interactive runs skip all once and record nothing; a failing item does not stop the job (continue, collect, report, job result failure); the target downloads from its own repos.
- folder_sync overlap (D-29): package jobs export their owned paths to folder_sync via the ADR-018 mechanism.

Consequences section: name the costly-to-reverse commitments — the manifest schema, the item identity scheme and the decision-file format are all shaped by D-01, so switching to file-level replication would replace the whole job core; and the decision files' location under `~/.config/pc-switcher/` means moving them later requires migrating user state on every machine.

Alternatives considered: file-level replication of the package databases (rejected — the package managers stay authoritative); a single `package_sync` job (rejected per D-15); a `--delete` file mirror of `/etc/apt` (rejected per D-11); source-cache reuse for offline installs (deferred per D-28).

Cite ADR-002, ADR-005, ADR-010, ADR-014, ADR-015 and ADR-018 where the position depends on them.
