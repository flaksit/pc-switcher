<objective>
Prove the tracer's claim against real machines: a package missing on the target really does travel source capture → target query → diff → plan → one batched review → `apt-get install` on the target.

Purpose: plan 02-03 wires the architecture and pins it with stubs. Stubs cannot tell you that `apt-mark showmanual` parses the way you assumed on a real Ubuntu 24.04 box, that sudo over the multiplexed SSH channel behaves, or that the review automation hook actually reaches the coordinator inside a real sync process. This plan is the difference between "the code is shaped correctly" and "the feature works".

It runs in the wave immediately after the tracer and before `snap_sync` and `flatpak_sync` are written, so an architectural mistake surfaces while exactly one manager exists.

Output: the phase's integration-test module, seeded with the tracer's end-to-end scenario.
</objective>

  <action>
Create `tests/integration/jobs/test_package_sync.py` with two tests. This file is the phase's integration module; plan 02-11 adds the whole-run contract tests to it later, so establish the fixture and teardown patterns those tests will reuse.

`test_apt_sync_installs_missing_package`:
- Pick a small, dependency-light apt package present on pc1 that can be removed from pc2 without collateral damage. Choose it at test time rather than hardcoding a guess: query pc1's `apt-mark showmanual`, intersect with pc2's installed set, and pick a candidate whose `apt-cache rdepends` shows no reverse dependencies among manually-installed packages. If no candidate is found, `pytest.skip` with a message naming what was searched — a skip that says why is honest; a silently-passing test is not.
- Remove the chosen package from pc2 (`sudo apt-get remove -y`) and assert it is gone from pc2's `apt-mark showmanual`.
- Run a real `pc-switcher sync pc2` from pc1 with `apt_sync: true`, driving the review non-interactively by setting `PCSWITCHER_PACKAGE_REVIEW_AUTOMATION` in the sync process's environment to a JSON mapping that applies exactly the chosen package's item id.
- Assert the package is back in pc2's `apt-mark showmanual` after the sync, and that the sync exited 0.

`test_apt_sync_dry_run_changes_nothing`:
- Same setup and the same automation mapping, but run the sync with `--dry-run`.
- Assert pc2's `apt-mark showmanual` output is byte-identical before and after, and that the sync exited 0. This is ADR-014's contract for a package job, and it is the cheapest possible guard against the plan/apply split leaking a mutating command into the planning phase.

Both tests restore pc2's package state in a fixture teardown regardless of outcome. Mark them with whatever integration marker `tests/integration/conftest.py`'s `pytest_collection_modifyitems` applies, and follow the existing tests' module/fixture scoping so the SSH connections are reused rather than re-established per test.

Do not add the non-interactive-skip-all, continue-on-failure, snap, flatpak or skip-always integration tests here — those belong to plan 02-11, once all three managers exist.
  </action>

## Trust Boundaries

| Boundary | Description |
| -------- | ----------- |
| test harness → real VM package state | The test removes and reinstalls a real package on the pc2 VM. |
| automation env var → the review's decisions | The hidden test hook pre-answers a human decision gate. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
| --------- | -------- | --------- | -------- | ----------- | --------------- |
| T-02-28 | Denial of Service | integration-test VM state | medium | mitigate | The subject package is selected by querying the VMs with a reverse-dependency check rather than hardcoded, and teardown restores it, so the test cannot leave pc2 missing something it needs. |
| T-02-29 | Repudiation | a silently-skipping test | medium | mitigate | Every candidate-selection failure path calls `pytest.skip` with a message naming what was searched, so a test that could not run is distinguishable from one that passed. |
| T-02-03 | Elevation of Privilege | `PCSWITCHER_PACKAGE_REVIEW_AUTOMATION` in CI | medium | mitigate | The variable only pre-answers decisions for items the diff already produced; it cannot widen scope, and it stays absent from `--help`, the config schema and user docs (D-26). |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No new package installs in this plan; the subject package is installed on the VM by the test through the system package manager under test. |
</threat_model>
