<objective>
Expand `apt_sync` from packages to the repository state those packages come from: sources, keys, pins and apt config, each as an item with the same three-way decision.

Purpose: replicating a package list without its repositories produces a target where half the installs fail with "no candidate". And doing it as a file mirror would wipe the target's own machine-specific sources — which is exactly why D-11 makes them items instead. The bundled-key rule (D-12) matters for the same reason: a repo that arrives without its key is a repo apt will refuse.

Output: five new apt item classes, captured, diffed, reviewed and converged in the order apt actually requires.
</objective>

  <behavior>
    - Converging a new repository whose key is also new issues the key write before the source write, and issues exactly one `apt-get update` after both, before any package install in the same run.
    - Converging a repository whose key already matches on the target issues no key write.
    - A key or source file is written by transferring the source's raw bytes to a staging path under the target user's home and promoting it with `sudo install` — the file's content on the target is byte-identical to the source's, and its owner, group and mode are root:root 0644.
    - Every `send_file` destination the job produces starts with the target user's home directory; none contains `/etc`.
    - The staging file is removed after a successful promotion, and also after a failed one, so a partial run leaves no readable copy of transferred key material lying in the user's cache.
    - A key is never fetched over the network: no converge command for a key item contains a URL.
    - Converging a `REMOVE` on a source item deletes that file and issues an `apt-get update` afterwards, and does not delete any other file under `/etc/apt`.
    - `apt-get update` runs at most once per job execution even when several repository items converged.
    - With `dry_run=True`, no staging transfer, no `sudo install`, no delete and no `apt-get update` reaches the target.
    - A failed key write is collected as a per-item failure and the dependent source item for that key is not written.
    - Transaction, happy path: when the metadata refresh succeeds, the backups taken for the repository group are discarded and the target's files are the newly written ones.
    - Transaction, rollback: when `apt-get update` exits non-zero, every file the group wrote is restored to its pre-run content, every file the group created that did not exist before is deleted, and only then is the failure reported.
    - Transaction, rollback verification: after a rollback the job re-runs `apt-get update` and reports in the failure summary whether apt recovered, so the user learns whether the target is usable rather than having to guess.
    - Transaction, isolation: a rollback of the repository group does not prevent the package items in the same run from being attempted (D-27).

  <action>
Extend `AptSyncJob.converge` to handle the four repository item classes, and add an ordering stage to the job's execute path.

Ordering: the base converge loop is item-by-item, but apt has a hard dependency chain — a repository whose key is missing fails verification, and a package from a repository whose metadata was never fetched has no candidate. Rather than making the base loop dependency-aware, have `AptSyncJob` sort its approved diffs before handing them to the loop: keys first, then pins and apt config, then sources, then a single `apt-get update` marker item, then packages. Document that this ordering is an apt fact rather than a general one, which is why it lives in the job and not in the shared core.

Represent the metadata refresh as a synthetic diff the job inserts when at least one source, key or pin item converged, so it flows through the same per-item logging, dry-run gate and failure collection as everything else instead of being a special case bolted onto the end. Insert it at most once.

File writes — stage, then promote. `RemoteExecutor.send_file` is plain SFTP running as the ordinary SSH user (see `executor.py` around lines 362-370); it has no sudo path, and the only existing caller writes under `$HOME`. It therefore cannot create a file in `/etc/apt` at all, and a temp path inside the destination directory is just as unwritable as the destination itself. The sequence per file is:
- Resolve the target user's home once per run the way `config_sync._copy_config_to_target` does — run `echo $HOME` on the target and cache the result — and derive a staging directory `<home>/.cache/pc-switcher/apt-staging/`. `mkdir -p` it. `~/.cache` is chosen because the shipped `home.filter` already excludes it from the mirror, so staging never becomes sync traffic.
- `send_file` the source file's bytes to `<staging>/<unique name>`. This is the only privileged-file path that touches SFTP, and its destination is inside the user's own home.
- Promote with a single `sudo install -o root -g root -m 0644 <staged> <destination>`. `install` sets ownership and mode atomically as part of the copy, so there is no window in which the file exists under `/etc/apt` with the wrong owner — which a `mv` followed by separate `chown` and `chmod` would leave open. 0644 for sources, pins and apt config, and also for keys under `keyrings/` and `trusted.gpg.d/`, which apt requires world-readable.
- Remove the staging file afterwards, in a `finally`-shaped path so a failed promotion does not leave transferred key material sitting in the user's cache.
- Preserve the original filename and therefore the original format extension. Use `shlex.quote` on every path.

Transactional convergence of the repository group. D-27 says a failing item does not stop the job — but applied naively to `/etc/apt` that means a failed `apt-get update` leaves the newly written sources and keys in place and every later apt operation on the target fails. Pre-sync btrfs snapshots exist, but automatic rollback is Phase 7, so until then this job must not be able to leave a machine whose package manager needs manual repair. Treat the key, source, pin and apt-config items plus their metadata refresh as one transaction:
- Before the first write in the group, back up every target file the group is about to change or delete: `sudo cp -a` each existing destination into a run-scoped backup directory under the same staging root, and record which destinations did not exist (those are rolled back by deletion, not restoration).
- Perform the group's writes and deletes in the established key → pin/config → source order.
- Run the synthetic `apt-get update` item.
- If it succeeds, discard the backup directory and continue to the package items.
- If it fails, restore: `sudo install` each backed-up file back to its destination with its original ownership and mode, `sudo rm -f` each destination that did not exist before, then run `apt-get update` once more purely to observe whether apt recovered. Log the restoration at ERROR with the original update's stderr, record every item in the group as a per-item failure naming that its change was rolled back, and include the post-rollback update result in the failure summary so the user is told whether the target's apt works. Then continue to the package items per D-27 — a package that needed the rolled-back repository will fail individually with apt's own no-candidate error, which is the honest report.
- Do all of this only when not in dry-run; in dry-run the whole group logs its intent and touches nothing.

Key/source coupling: before writing a source item, check that every keyring path it references either already matches on the target or is itself among the approved-and-successfully-converged key items in this run. If not, skip the source write, log at ERROR naming the missing keyring, and collect it as a per-item failure (D-27) — a repository written without its key is a repository apt will refuse on every subsequent operation, so writing it anyway makes the target worse than leaving it alone.

Removals: delete the single named file with `sudo rm -f <quoted path>`. Never use a recursive or glob delete, and never run rsync with `--delete` against `/etc/apt` — the target's own machine-specific repositories must survive (D-11).

Extend `tests/unit/jobs/test_apt_sync.py` with the behavior bullets, asserting on the exact ordered command sequence the stub target executor received.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| source `/etc/apt` → target `/etc/apt` | Repository definitions and signing keys cross machines and expand what the target will trust and install from. |
| transferred keyring bytes → apt's verification | The key the target uses to verify every future package from that repository. |
| `/etc/apt` file paths → root shell commands | Filenames from a directory listing are interpolated into root-privileged write and delete commands. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-03 | Spoofing | signing-key provisioning | high | mitigate | Keys are copied byte-for-byte from the source's own verified keyring; no converge command contains a vendor URL and no key is regenerated or re-derived. apt's own verification on `update`/`install` performs the actual cryptography (D-12, ASVS V6). |
| T-02-04 | Tampering | trust-set expansion | high | mitigate | Every source and key item is its own named review entry with the three-way decision; adding a repository to the target is always a decision the user saw, never a side effect of installing a package (D-11). |
| T-02-15 | Tampering | `/etc/apt` deletion | high | mitigate | Removals delete one named file via `rm -f`; there is no recursive delete and no rsync `--delete` against `/etc/apt`, so the target's machine-specific repositories survive (D-11). |
| T-02-16 | Denial of Service | repo written without its key | medium | mitigate | A source item whose keyring is missing or whose key write failed is skipped and reported, rather than written to produce a target where every apt operation fails signature verification. |
| T-02-34 | Denial of Service | failed `apt-get update` after `/etc/apt` writes | high | mitigate | The repository/key/pin/config group is transactional: files are backed up before the writes, and a failed metadata refresh restores them and re-probes apt before reporting, so D-27's continue-and-report model cannot leave the target's package manager unusable while automatic rollback remains a Phase 7 feature. |
| T-02-35 | Elevation of Privilege | privileged file promotion | high | mitigate | Bytes are staged under the target user's own home and promoted with one `sudo install -o root -g root -m 0644`, which sets ownership and mode as part of the copy — there is no interval in which a file exists under `/etc/apt` owned by the SSH user, which a `mv` plus separate `chown` would create. The staging copy is deleted on both success and failure. |
| T-02-01 | Tampering | filename interpolation into root commands | high | mitigate | `shlex.quote()` on every path in every write, move, chown, chmod and delete command (ASVS V5). |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No package installs in this plan. |
</threat_model>
