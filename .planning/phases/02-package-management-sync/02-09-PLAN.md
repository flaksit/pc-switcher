<objective>
The `flatpak_sync` job: refs, origins and the user/system scope split, plus remote provisioning in the order flatpak requires.

Purpose: flatpak is the one manager where scope is part of identity. The same runtime can be installed in both user and system scope on one machine, and `flatpak install` fails outright if the remote is not configured in the scope being installed into. Getting either wrong produces a target that looks converged and is not.

Output: the third manager job, built on the proven core, ready for wiring in plan 02-10.
</objective>

  <action>
Create `src/pcswitcher/jobs/flatpak_sync.py` following the structure of `apt_sync.py` and `snap_sync.py`.

Module docstring: state that scope is part of a flatpak item's identity, that the same ref in user and system scope is two items rather than one, that the OSTree store stays authoritative for its own state (D-01), and that `~/.var/app` — the per-application user data — stays folder_sync's territory while `~/.local/share/flatpak` is this job's (D-29). Note that several runtimes on the reference machine are installed in both scopes and that normalising that is deliberately out of scope: it is a change to the machines, not a sync feature.

`class FlatpakSyncJob(PackageSyncJob)` with `name: ClassVar[str] = "flatpak_sync"`, `manager_name = "flatpak"`, and a `CONFIG_SCHEMA` matching the sibling jobs' minimal shape. Implement only the abstract hooks and inherit `plan()`, `accept_review()`, `apply()` and `execute()` unchanged. Do not override `execute()` and do not call `review_items` from this module — the single cross-manager review belongs to `PackagePhaseCoordinator`. Keep capture and query free of any mutating command; they run in the read-only planning phase before anything is approved.

Capture and query, both machines:
- `flatpak list --app --columns=application,version,origin,installation` — one call yields application, version, origin remote and scope together. Parse it tab-separated; unlike `snap list` this invocation names its columns explicitly in the command, so the output shape is what the command asked for.
- `flatpak remotes --columns=name,url`, run once per scope (`--user` and `--system`), since flatpak tracks per-installation remote lists even when the URL is identical.
- Emit `FlatpakItem` and `FlatpakRemoteItem` imported from `package_items.py`, where plan 02-05 defined them: `FlatpakItem` with `application: str`, `version: str`, `origin: str`, `scope: Literal["user", "system"]` and identity `flatpak:ref:<scope>:<application>`; `FlatpakRemoteItem` with `name: str`, `url: str`, `scope: Literal["user", "system"]` and identity `flatpak:remote:<scope>:<name>`. Scope is already inside both identity strings, which is what makes the two-scope case fall out of the generic diff engine with no special casing here. This plan populates those shapes and leaves `package_items.py` untouched — the module's item shapes are frozen after wave 4, which is what lets this plan run alongside 02-07 and 02-08.
- Version is captured for reporting only. Per D-04 versions float; a version difference on a flatpak ref is a report-only diff, not a change to force.

Converge:
- Add remote: `flatpak remote-add --if-not-exists <scope flag> <quoted name> <quoted url>`.
- Install ref: `flatpak install -y <scope flag> <quoted origin> <quoted application>`. The scope flag is `--user` or `--system` taken from the item; `--system` needs `sudo`, `--user` does not.
- Remove ref: `flatpak uninstall -y <scope flag> <quoted application>`.
- Remove remote: `flatpak remote-delete <scope flag> <quoted name>`.
- Every interpolated value goes through `shlex.quote`.

Ordering: give the job its own pre-loop sort placing remote diffs before ref diffs, the same job-local ordering stage `apt_sync` uses for keys-before-sources, and for the same reason — the dependency is a fact about this ecosystem, not about the shared core. Before converging a ref, check its origin remote is present on the target in the ref's scope or was converged in this run; if not, skip it and collect a per-item failure naming the missing remote, rather than issuing an install that flatpak will reject.

`validate()`: sequential checks appending to an `errors` list — `flatpak --version` on source and target, and `sudo -n true` on the target only when at least one system-scope item is in play (user-scope operations need no root, ASVS V4). Flatpak's absence on a machine is a reported validation error naming that flatpak is not part of a default Ubuntu 24.04 install, not an exception.

`describe_first_sync_scope()`: `scope_items=["installed flatpak refs (per user/system scope)", "configured flatpak remotes (per scope)"]`, `mechanism="flatpak install/uninstall/remote-add per item, after review"`.

`flatpak_sync_exclude_paths() -> list[Path]` as a module-level function resolving against `Path.home()` at call time. Return `~/.local/share/flatpak` only. Document that `~/.var/app` is deliberately not returned: it is per-application user data that folder_sync should keep mirroring, and D-17's ordering exists precisely so `flatpak install` creates the store before folder_sync lands that data on top.

Tests in `tests/unit/jobs/test_flatpak_sync.py` cover every behavior bullet with stub executors returning realistic tab-separated output including a both-scopes runtime and a `flathub` remote present in both scopes.

<threat_model>
## Trust Boundaries

| Boundary | Description |
| -------- | ----------- |
| flatpak CLI output → command construction | Application ids, remote names and URLs parsed from CLI output are interpolated into install commands. |
| source remote list → target's installable software sources | Adding a remote expands what the target machine will install executable software from. |
| user scope vs system scope | The two scopes have different privilege requirements and different blast radius. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
| --------- | -------- | --------- | -------- | ----------- | --------------- |
| T-02-22 | Tampering | remote provisioning | high | mitigate | Every remote is its own named review item with the three-way decision; a remote is never added as an implicit side effect of installing a ref (D-11, D-14). |
| T-02-23 | Elevation of Privilege | scope selection | high | mitigate | The scope flag is taken from the item's own captured scope, and scope is part of item identity, so a user-scope app cannot silently become a system-scope (root-installed) one; `sudo` is requested only when a system-scope item is actually in play (ASVS V4). |
| T-02-24 | Denial of Service | install with an unconfigured remote | medium | mitigate | A ref whose origin remote is absent in its scope is skipped with a named per-item failure rather than issuing an install flatpak will reject, so the run's report says what is actually wrong. |
| T-02-01 | Tampering | application id / remote name / URL interpolation | high | mitigate | `shlex.quote()` on every interpolated value in every flatpak command (ASVS V5). |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No package installs in this plan. |
</threat_model>
