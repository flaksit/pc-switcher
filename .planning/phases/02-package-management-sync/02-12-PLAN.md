  <action>
Add a Package Sync section to `docs/configuration.md` covering:

- The three jobs, their `sync_jobs:` keys, that all three ship disabled, and that they run before `folder_sync` so applications exist before their data lands.
- What each job covers: `apt_sync` handles the manually-installed package set plus repositories, signing keys, pins and apt config under `/etc/apt`; `snap_sync` handles snaps with their channel and revision; `flatpak_sync` handles refs and remotes per user/system scope.
- The batched review: what it shows, that ONE review covers every enabled package job at once rather than one review per manager, that it appears before anything is changed on any manager, that items are grouped by manager and by action, that installs and removals are separate groups, that removals start unticked, and that each item offers apply, skip once, or skip always.
- What the review shows about apt beyond the packages themselves: that apt may need to remove or change other packages to satisfy a request, that those collateral effects are shown in the review, and that an approved item whose real transaction would exceed what was reviewed is refused and reported rather than executed.
- Machine-specific packages: use that phrasing throughout, not "exclusion". Explain that marking a package machine-specific records it in that machine's own decision file, that the file is never synced, that the entry makes the item inert on that machine whether it is source or target, and how to un-mark something by deleting the entry. Give the file path and point at the shipped example file.
- Install snippets: what they are for (things no package manager can reinstall), where the registry lives, that it IS synced — carried alongside `config.yaml` by config sync, with its own diff-and-confirm prompt when the two machines' registries differ — because install knowledge belongs to the package rather than the machine, that a snippet is stored and replayed verbatim and never interpreted, and that it must run non-interactively because no stdin is available. Reproduce the same worked shape the capture prompt shows so the guidance is identical in both places. State plainly that while any unreproducible item is left with neither a snippet nor a machine-specific mark, the package job reports failure at the end of every interactive run — that is deliberate, and it is how the tool keeps asking rather than letting the gap disappear.
- The folder_sync interaction: enabling `snap_sync` or `flatpak_sync` automatically excludes that ecosystem's store from the mirror, that this cannot be overridden by a user filter rule, and that any hand-written rules for those paths in a personal filter file can now be deleted.
- Versions: that the tool installs by name and lets versions float to what each machine's repositories offer, that a version difference is reported rather than forced, and that deliberate pinning replicates because `preferences.d` files sync as items.
- A short non-interactive note: a run without a TTY changes nothing, records nothing, and reports everything.

Do not mention the review-automation environment variable anywhere.

In `README.md`, extend wherever sync jobs are introduced with a one-or-two-line entry per package job, pointing at the configuration document for detail.

  <how-to-verify>
Confirm the phase's three roadmap success criteria on your own two machines, not on the test VMs.

Setup: enable all three package jobs on the source by setting `apt_sync: true`, `snap_sync: true` and `flatpak_sync: true` in `~/.config/pc-switcher/config.yaml`.

1. Preview first: `pc-switcher sync <target> --dry-run`
   - Check that ONE review appears covering all three managers, not three separate reviews — this is the phase's central promise and the thing easiest to get subtly wrong.
   - Check the review appears before anything else and shows every difference grouped by manager and by action.
   - Check installs and removals are visibly separate groups and that the removal group names removal, not "apply".
   - If apt would need to remove or change anything beyond what you ticked, check that those collateral effects appear in the review as their own entries rather than only surfacing at apply time.
   - Check version mismatches show both versions, and that held or pinned packages are called out.
   - Check the four apt packages with no repository candidate appear as unreproducible items needing a resolution.
   - Check nothing changed on the target: run `apt-mark showmanual | wc -l`, `snap list | wc -l` and `flatpak list | wc -l` on the target before and after and compare.

2. Real run: `pc-switcher sync <target>`
   - Tick a few installs, leave the removals unticked, mark one genuinely machine-specific package as skip-always, and author a snippet for one unreproducible item.
   - After it finishes, on the TARGET: check the ticked packages are installed (`apt-mark showmanual`, `snap list`, `flatpak list`), and that nothing you left unticked was removed.
   - On the SOURCE: check `~/.config/pc-switcher/apt.decisions.yaml` now names the package you marked machine-specific.
   - Check `~/.config/pc-switcher/package-snippets.yaml` contains the snippet you authored, verbatim.
   - On the TARGET: check `~/.config/pc-switcher/package-snippets.yaml` also exists and matches — the registry is synced, and this is the step that proves it actually travels.
   - On the TARGET: check no `*.decisions.yaml` file arrived from the source. Those are machine-local and must never travel.

3. Second run: `pc-switcher sync <target> --dry-run`
   - Check the item you marked machine-specific produces no difference at all this time.
   - Check the item you gave a snippet to is no longer listed as needing a resolution.

4. Auto-refresh guard: on both machines run `snap get system refresh.hold` and confirm it reads the same as it did before any of this.

5. Mirror interaction: confirm `~/.var/app` content still syncs and that `~/.local/share/flatpak` is no longer being mirrored, then delete the now-redundant flatpak and snap lines from `~/.config/pc-switcher/home-janfr.filter` and re-run a dry-run to confirm nothing changed.
  </how-to-verify>

| T-02-30 | Elevation of Privilege | snippet-authoring guidance | medium | mitigate | The documented snippet shape matches the capture prompt's, and both state the non-interactive requirement; divergent guidance would lead users to author snippets that hang a root-privileged sync. |
