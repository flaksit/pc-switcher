<objective>
The `snap_sync` job: name, channel and revision convergence through snapd's own verbs, plus the path-export folder_sync needs to stop fighting it.

Purpose: D-06 sets a constraint that is easy to violate accidentally — both machines should land on the same revision, but neither should stop updating. Research found the trap live: `snap refresh --hold` with no arguments is a mutating command that silently sets an indefinite global hold. This job must converge without ever going near it.

Output: a complete third-party-manager job built on the proven core, ready to be wired into config and folder_sync in plan 02-10.
</objective>

  <action>
Create `src/pcswitcher/jobs/snap_sync.py` following the structure of `apt_sync.py`.

Module docstring: state that this job owns snap presence, tracking channel and revision, that the snapd store stays authoritative for its own state (D-01), and — prominently — that the no-argument `snap refresh --hold` form is a mutating command that sets an indefinite global auto-refresh hold, so hold state is inspected only through the read-only `snap get system refresh.hold`. Research discovered this live on a real machine; the warning belongs in the code, not only in a plan.

`class SnapSyncJob(PackageSyncJob)` with `name: ClassVar[str] = "snap_sync"`, `manager_name = "snap"`, and a `CONFIG_SCHEMA` matching `apt_sync`'s minimal shape. Implement only the abstract hooks — `capture_source_items`, `query_target_items`, `converge` — and inherit `plan()`, `accept_review()`, `apply()` and `execute()` unchanged. Do not override `execute()` and do not call `review_items` from this module: the single cross-manager review belongs to `PackagePhaseCoordinator`, and a job that reviewed its own items would reintroduce exactly the per-manager review D-24 forbids. Keep capture and query free of any mutating command — they run during the read-only planning phase, before the user has approved anything.

Capture and query, both machines, same commands (source via `self.source`, target via `self.target` with `login_shell=False`):
- `snap list --all`. There is no `--json` flag on the shipped snap CLI, so parse the columnar output: read the header line, build a name-to-index mapping from it, and pull `Name`, `Rev`, `Tracking` and `Notes` by that mapping. Never index by fixed character offsets or assumed order. A snapd version that reorders or adds columns must not silently corrupt the manifest — that is the whole reason for header-driven parsing.
- Skip lines whose `Notes` mark them as a disabled older revision; only the active revision is the item.
- Emit a `SnapItem` imported from `package_items.py`, where plan 02-05 defined it with `name: str`, `channel: str`, `revision: str` and identity `snap:<name>`. Use `ItemClass.SNAP_CHANNEL` only for the diff detail on a channel-only change, never as a standalone item. This plan populates that shape and leaves `package_items.py` untouched — the module's item shapes are frozen after wave 4, which is what lets this plan run alongside 02-07 and 02-09.

Converge:
- Install (missing on target): `sudo snap install --revision=<quoted revision> <quoted name>`, then `sudo snap switch --channel=<quoted channel> <quoted name>` when the tracking channel differs from snapd's default for that snap. Installing at an explicit revision is the mechanism that converges without a hold (D-06).
- Change (different revision): `sudo snap refresh --revision=<quoted revision> <quoted name>`, plus the same channel switch when the channel also differs.
- Change (same revision, different channel): the channel switch alone.
- Remove (extra on target): `sudo snap remove <quoted name>`. Do not pass a purge flag — snap's purge discards the snapshot snapd takes on removal, and that snapshot is the user's only recovery path if the removal was a mistake; note that reasoning in a comment.
- Every interpolated value goes through `shlex.quote`.

`validate()`: sequential checks appending to an `errors` list — `snap version` on source and target, `sudo -n true` on the target, and a read-only `snap get system refresh.hold` on both machines whose result is logged at FULL as informational context (a pre-existing hold explains why a revision will not converge) and never acted on.

`describe_first_sync_scope()`: `scope_items=["installed snaps (name, channel, revision)"]`, `mechanism="snap install/refresh/remove per item, after review"`.

`snap_sync_exclude_paths() -> list[Path]` as a module-level function, not a constant, resolving against `Path.home()` at call time exactly like `vscode_state_exclude_paths()`. Return the per-revision data directories `~/snap/<app>/<revision>` — enumerate them from the filesystem, since unlike VS Code's fixed relpath list the revision directories are dynamic. Deliberately do NOT return `~/snap/<app>/common` or `~/snap/<app>/current`: `common` is revision-independent user data that folder_sync should keep mirroring, and `current` is a symlink snapd maintains. Document that distinction in the function's docstring — it is the whole reason this export is not simply `~/snap`.

Tests in `tests/unit/jobs/test_snap_sync.py` cover every behavior bullet with stub executors returning realistic `snap list --all` output, including a column-reordered variant and a disabled-revision line.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| snapd CLI output → command construction | Snap names, revisions and channels parsed from columnar output are interpolated into root-privileged commands. |
| source orchestrator → target snapd | Install, refresh and remove commands run as root on the target. |
| convergence action → target's update policy | A convergence mechanism can accidentally alter the machine's standing auto-refresh behaviour. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-05 | Denial of Service | snap auto-refresh policy | high | mitigate | Convergence uses `snap install --revision` / `snap refresh --revision` only; hold state is read through `snap get system refresh.hold` and never written. A test collects every command the job issues and asserts none of them sets a hold, so a future edit cannot reintroduce it silently (D-06, RESEARCH Pitfall 1). |
| T-02-20 | Tampering | columnar output parsing | medium | mitigate | Column positions are resolved from the header row rather than assumed, so a snapd column reorder produces a parse failure or correct values, never a silently wrong revision that drives a wrong `--revision` install. |
| T-02-21 | Tampering | snap removal | medium | mitigate | Removal omits the purge flag, so snapd's automatic pre-removal snapshot survives and a mistaken removal remains recoverable. |
| T-02-01 | Tampering | snap name/revision/channel interpolation | high | mitigate | `shlex.quote()` on every interpolated value in every snap command (ASVS V5). |
| T-02-04 | Access Control | target snap commands | high | mitigate | `validate()` checks `sudo -n true` on the target before any converge runs (ASVS V4). |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No package installs in this plan. |
</threat_model>
