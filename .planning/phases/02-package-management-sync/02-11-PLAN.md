  <action>
Add six tests to `tests/integration/jobs/test_package_sync.py`, alongside the two the tracer's proof plan (02-13) already put there.

`test_non_interactive_skip_all` — run a sync from pc1 to pc2 with a genuinely non-interactive process (no TTY on stdin or stdout, and without setting the automation env var), with a package diverged in each direction. Assert: the sync completes, pc2's package set is byte-identical to what it was before the run, no decision file exists on either machine that did not exist before, and the run's output names each unresolved item. The evidence is pc2's own package manager plus the decision-file paths, not pc-switcher's log narrative.

`test_continue_on_item_failure` — the failing item must genuinely enter an apply path, which rules out the obvious choice: a package name that resolves to nothing is classified `REPO_UNAVAILABLE` with action `REPORT_ONLY` (plan 02-05), and `REPORT_ONLY` items short-circuit before touching the target, so such an item never reaches converge and would prove nothing about D-27.

Use a deliberately failing install snippet instead. It is the cleanest controlled failure available: it runs through the same converge dispatch as every other item, its exit code alone decides success (D-20), and it cannot destabilise apt, snap or flatpak state on the VM because it never invokes them.
- Set up: on pc1, write a snippet registry entry whose body is a script exiting non-zero with a recognisable code (something like `echo "deliberate integration-test failure" >&2; exit 42`) keyed to an unreproducible item id the run will produce. Author it directly into pc1's `~/.config/pc-switcher/package-snippets.yaml` in the fixture rather than through the review, so the test does not depend on the interactive capture path.
- Arrange two viable apt package installs around it, selected with the same query-the-VMs approach the other tests use, so the failing item sits between two items that must both still be attempted.
- Drive the review via the automation env var to apply all three items.
- Assert: both viable packages are installed on pc2 afterwards (queried from pc2's own `apt-mark showmanual`), proving the item after the failure was still processed; the snippet item is named in the run's summary with its stderr and its non-zero exit; and the sync's exit code is non-zero.
- Teardown removes the snippet entry from pc1's registry and restores pc2's packages.

`test_snap_revision_converges_without_hold` — record `snap get system refresh.hold` on both VMs, diverge one snap's revision, run the sync applying that item, then assert pc2's `snap list` shows the source's revision and that `snap get system refresh.hold` on both VMs is unchanged. Skip with a message naming what was searched if no snap can be safely diverged.

`test_flatpak_installs_into_source_scope_after_remote` — remove a flatpak remote and one ref from pc2, run the sync applying both items, then assert pc2 has the remote in the expected scope and the ref installed in the same scope the source item carried. Assert the remote existed before the ref install by checking the ordering in the run's per-item log, since the remote's presence afterwards alone would not distinguish the two orders.

`test_skip_always_is_inert_in_both_roles` — run a sync driving the review to record skip-always on one source-held item, assert the decision file on pc1 now names it, then run a second sync and assert that item produces no diff at all. Then reverse the direction (sync pc2 to pc1) and assert the same item still produces no diff, proving the entry is inert in both roles. Clean up the decision file in teardown so the test is re-runnable.

`test_all_managers_diff_before_any_applies` — the whole-run proof of D-24, and the reason the coordinator exists. Enable at least two package jobs (`apt_sync` and `snap_sync`, or `flatpak_sync` if a snap subject cannot be safely diverged) and diverge one item in each manager so both have something to apply. Run the sync applying both items. Assert from the run's own per-item log ordering that every enabled manager's capture and diff completed before the first mutating command of any manager — the property is about ordering, so the log is the right evidence here, and this is the one place the plan's "assert against the package manager, not the log" rule does not apply. Also assert the end state from the package managers themselves: both diverged items converged. Skip with a naming message if a second manager has no safely divergeable subject.

Every test restores VM state in teardown regardless of outcome. Reuse the tracer's candidate-selection approach — query the VMs for a suitable subject rather than hardcoding a package name that may not exist — and skip with a naming message when none is found.

## Trust Boundaries

| Boundary | Description |
| -------- | ----------- |
| test harness → real VM package state | Tests mutate real package managers on the pc1/pc2 VMs. |
| automation env var → the review's decisions | The hidden test hook pre-answers a human decision gate. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
| --------- | -------- | --------- | -------- | ----------- | --------------- |
| T-02-28 | Denial of Service | integration-test VM state | medium | mitigate | Every test restores what it changed in teardown, and candidate subjects are selected by querying the VMs (with a reverse-dependency check) rather than hardcoded, so a test cannot uninstall something the VM needs. |
| T-02-29 | Repudiation | a silently-skipping test | medium | mitigate | Every candidate-selection failure path calls `pytest.skip` with a message naming what was searched, so a test that could not run is distinguishable from one that passed. |
| T-02-03 | Elevation of Privilege | `PCSWITCHER_PACKAGE_REVIEW_AUTOMATION` in CI | medium | mitigate | The variable only pre-answers decisions for items the diff already produced; it cannot widen scope. It remains absent from `--help`, the config schema and user docs (D-26). |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No new package installs in this plan; test-subject packages are installed on the VMs by the tests themselves through the system package managers under test. |
</threat_model>
