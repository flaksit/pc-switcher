<objective>
Give every machine a permanent, machine-local record of what it considers machine-specific, and make sure that record can never travel.

Purpose: D-19's whole argument for scanning aggressively is that a finding produces noise exactly once and then never again. That only holds if "skip always" is durable. And D-08's one rule — an item recorded on machine M is inert on M in both roles — is what collapses "this is my hardware driver, don't share it" and "don't put that here" into a single mechanism instead of two.

Output: a decision-file store, its example file, and the non-overridable folder_sync exclusion that keeps it machine-local.
</objective>

Create `src/pcswitcher/jobs/package_state.py`.

Module docstring: state that these files are the ONLY pc-switcher state that is deliberately per-machine and never synced, that an entry means "inert on this machine in both roles" (D-08), and that the file for a given entry is chosen by which machine holds the item (D-08a) — so writes go through whichever executor is local to that machine, never through a local `pathlib` write for the target side (ADR-002: the target has no direct filesystem access from here).

Public surface:
- `DECISION_FILE_RELPATH_TEMPLATE = ".config/pc-switcher/{manager}.decisions.yaml"` — derive the directory portion from `config_sync.CONFIG_REMOTE_DIR` rather than writing the literal a second time, matching the CR-01 precedent that folder_sync already follows.
- `DECISION_FILE_GLOB_RELPATH = ".config/pc-switcher/*.decisions.yaml"` — the single value `folder_sync` consumes for its exclusion, so the exclusion and the store cannot drift apart.
- A frozen `DecisionEntry` dataclass: `item_id: str`, `item_class: ItemClass`, `label: str`, `reason: str | None`, `recorded_at: str` (ISO-8601 UTC).
- `class DecisionFile` holding a manager name and an executor, with `async def load() -> dict[str, DecisionEntry]` and `async def record(entry: DecisionEntry) -> None`.

Implementation notes for `DecisionFile`:
- `load()` runs `cat <quoted path> 2>/dev/null` through its executor and parses the result with `yaml.safe_load`. A non-zero exit, empty output or a parse failure yields an empty mapping; a parse failure additionally logs a WARNING naming the path. An absent file is normal on a fresh machine and must not be an error.
- `record()` reads the current content, merges the new entry by `item_id` (last write wins on the reason, no duplicate keys), serialises with `yaml.safe_dump`, and writes atomically: `mkdir -p` the directory, write to `<path>.pcswitcher-tmp`, then `mv -f` into place — the same atomic-replace-within-the-same-directory shape `vscode_state_sync._sync_editor` uses. Transfer the serialised bytes as an argv-quoted string through the executor rather than assuming local filesystem access, so the same code path works for the target.
- File format: a top-level `machine_specific:` mapping of item id to `{item_class, label, reason, recorded_at}`, plus a leading comment block the writer regenerates each time explaining that this file is machine-local, never synced, and that removing an entry makes the item eligible again.

Also add `async def filter_inert(items, decisions)` as a module-level pure function taking a sequence of items with `item_id` and a decision mapping, returning the items whose ids are absent from the mapping. Keeping it pure and module-level means both the source-capture and target-query sides use exactly one implementation of "inert".

Create `src/pcswitcher/machine-packages.example.yaml`: a commented example decision file with two illustrative entries (a printer/scanner driver and a machine-specific vendor package) shown as comments rather than active entries, so copying it produces an empty-but-documented file. State in its header that it is an example, that the real file accumulates from recorded skip-always decisions, and that pc-switcher never writes defaults into it. Do not add any of its content to `default-config.yaml` and do not hardcode any of it in Python (D-10).

Tests in `tests/unit/jobs/test_package_state.py` cover every behavior bullet using stub executors that record the commands they receive, asserting both the command shape (atomic temp-then-move) and the parsed round-trip.

Note in the same comment that this exclusion is unconditional rather than gated on the package jobs being enabled — the ADR-018 VS Code exclusion is gated on its job, but a decision file must never travel even on a machine where the package jobs are currently off, or toggling a job off and running one sync would leak it.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| decision file → folder_sync mirror | A machine-local state file sits inside the mirrored `/home` tree and must never cross to a peer. |
| decision-file content → YAML parser | File content is user-editable and parsed on both machines. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-08 | Information Disclosure | `~/.config/pc-switcher/*.decisions.yaml` | medium | mitigate | Unconditional, GLOBAL-FIRST folder_sync exclusion that no user `+` rule can override, plus confirmation that config_sync transfers only `config.yaml` (D-09, ADR-017 precedent). |
| T-02-10 | Tampering | decision-file YAML parsing | medium | mitigate | Parse with `yaml.safe_load` (never `yaml.load`); a malformed file degrades to "no permanent decisions" with a WARNING rather than raising or being partially interpreted. |
| T-02-11 | Tampering | atomic decision-file write | medium | mitigate | Write to a sibling temp path and `mv -f` into place, so an interrupted or failed write cannot leave a truncated file that silently drops previously-recorded machine-specific items. |
| T-02-01 | Tampering | decision-file path interpolation | high | mitigate | `shlex.quote()` on every path and serialised payload reaching a command string, on both the local and remote executor paths (ASVS V5). |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No package installs in this plan. |
</threat_model>
