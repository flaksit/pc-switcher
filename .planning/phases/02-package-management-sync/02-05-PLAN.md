<objective>
Expand the tracer's one direction into the full diff taxonomy: removals, version mismatches, held and pinned packages, repo-unavailable and unreproducible items — all as classes inside the one batched review, not as a second reporting mechanism.

Purpose: this is where REQ-conflict-detection-no-resolution actually lands. D-25 is explicit that "conflicts and version mismatches" are diff classes in the same review, and success criterion 3's "reported before any destructive change" is satisfied because the whole review precedes every change. This plan makes the taxonomy real and makes removals structurally impossible to confuse with installs.

Output: a complete diff engine, a correct deb version comparator, review grouping that names what will actually happen to each item, and the complete item-shape registry the three manager jobs import.
</objective>

  <action>
Extend `src/pcswitcher/jobs/package_items.py`.

Add `async def compare_deb_versions(executor, left: str, right: str) -> int` returning a negative number, zero or a positive number in the `sorted`-comparator convention. Implement it by shelling out to `dpkg --compare-versions <left> lt <right>` and `dpkg --compare-versions <left> gt <right>` through the supplied executor, with `shlex.quote` on both operands, and short-circuit to zero when the two strings are byte-identical so the common case costs no subprocess. Give it a docstring stating why it is not hand-rolled: Debian version ordering has epoch, tilde and revision tie-breaking rules that are neither lexicographic nor PEP 440, and only dpkg's own comparator is correct.

Add a `HoldPinFact` frozen dataclass carrying `mechanism: Literal["hold", "pin"]`, `package: str`, and `source_ref: str` (the `apt-mark showhold` origin or the `preferences.d` filename). Document that hold and pin are two different mechanisms — a hold is dpkg selection state under `/var/lib/dpkg`, a pin is a priority preference under `/etc/apt` — that both surface under one review category but must stay distinguishable, and that reading only one of the two sources silently misses half the held packages.

Add `def build_detail(...)` helpers producing the review's `detail` string per diff class, so the phrasing lives in one place: version mismatch shows both versions with the machine each came from; held-or-pinned shows the mechanism and its origin; repo-unavailable names that the target's repositories offer no candidate.

Add the four remaining item dataclasses, so `package_items.py` holds every item shape the manager jobs will populate before any of those jobs is written. Each is frozen and carries `item_id` and `label()` following `AptPackageItem`'s convention, against an `ItemClass` member the tracer already declared:

- `SnapItem`: `name: str`, `channel: str`, `revision: str`. Identity `snap:<name>`. Channel is a field of the snap item rather than an item class of its own, because a channel with no snap has no meaning; `ItemClass.SNAP_CHANNEL` stays reserved for the diff detail on a channel-only change and never becomes a standalone item. State that in a comment on the class.
- `FlatpakItem`: `application: str`, `version: str`, `origin: str`, `scope: Literal["user", "system"]`. Identity `flatpak:ref:<scope>:<application>`.
- `FlatpakRemoteItem`: `name: str`, `url: str`, `scope: Literal["user", "system"]`. Identity `flatpak:remote:<scope>:<name>`.
- `UnreproducibleItem`: `origin: Literal["apt-no-candidate", "unowned-path"]`, `identifier: str` (a package name or a filesystem path), `label: str`. Identity `unreproducible:<origin>:<identifier>`.

Scope belongs inside both flatpak identity strings, and `origin` inside the unreproducible one, for the same reason: it makes the same-name-different-context case fall out of the generic diff engine as two items with no special casing in any job.

Document in the module docstring that `package_items.py` is the phase's single registry of item shapes — the manager jobs in plans 02-07, 02-08 and 02-09 import these and add none of their own. Those three plans are written in parallel, and a shape defined in three places is a shape that drifts.

Tests in `tests/unit/jobs/test_package_items.py` cover every behavior bullet. For `compare_deb_versions`, use a stub executor asserting the exact commands issued, plus — if the developer machine has dpkg — one test that exercises the real binary so the stub's assumptions cannot drift from reality; guard that one with a skip when `dpkg` is absent.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| diff engine → destructive target commands | A misclassified diff turns an install into a removal of working software. |
| version strings → comparison | Version strings from two machines' package managers decide whether a change is proposed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-12 | Tampering | removal grouping in the review | high | mitigate | Removals occupy their own `(manager, action)` group with an action-specific verb title and unchecked-by-default entries, so a bulk confirm cannot silently delete software (D-07, D-24). |
| T-02-13 | Tampering | deb version comparison | high | mitigate | Ordering is delegated to `dpkg --compare-versions`; a hand-rolled comparator would rank `2:1.0` below `10.0` and could propose a downgrade as an upgrade. |
| T-02-14 | Denial of Service | hold/pin detection | medium | mitigate | Both `apt-mark showhold` and `preferences.d` are read; reading only one would silently miss half the held packages and propose changes the target will refuse, producing a run where every item fails. |
| T-02-01 | Tampering | version and package-name interpolation | high | mitigate | `shlex.quote()` on both operands of every `dpkg --compare-versions` call and on every package name in a removal command (ASVS V5). |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No package installs in this plan. |
</threat_model>
