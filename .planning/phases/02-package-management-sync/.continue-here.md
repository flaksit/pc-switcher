# BLOCKING CONSTRAINTS — Read Before Anything Else

> These are not suggestions. Each was discovered through actual failure this session.
> Acknowledge each one explicitly before proceeding.

- [ ] CONSTRAINT: A decision record is evidence, not law — when two locked decisions conflict, ask, do not pick. `02-CONTEXT.md` D-24 read as "one batched review across all managers" while D-15 said the jobs are independent. The orchestrator resolved that tension by declaring D-24 LOCKED to a cross-AI reviewer, and then had the planner build a `PackagePhaseCoordinator` to satisfy it. The user had never asked for a cross-manager review. Structural mitigation: before treating a CONTEXT decision as binding on a design of this size, grep the DISCUSSION-LOG for what the user was actually asked and what they actually chose; if the decision conflicts with another decision, stop and ask.
- [ ] CONSTRAINT: The orchestrator does not author implementation plans. Late in this session the orchestrator hand-wrote a five-wave rework plan with file-ownership analysis and per-agent assignments — the `gsd-planner`'s job. It produced a plan that changed code while leaving the wrong decisions in `02-CONTEXT.md`, which would have let the next planner rebuild the removed coordinator. Structural mitigation: rework goes through `/gsd-plan-phase`, and the entry point for wrong-decision feedback is CONTEXT.md, never a hand-rolled task list.

**Do not proceed until both boxes are checked.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Decision-record escalation | Treating a discuss-phase paraphrase as a locked user requirement, then building architecture on it. Produced `package_phase.py` plus 457 lines of tests for a coordinator the user never wanted. | blocking | Check DISCUSSION-LOG for the actual question and answer; surface conflicts between decisions to the user rather than resolving them silently. |
| Orchestrator-authored plans | Orchestrator writing waves, file-ownership maps and agent assignments inline instead of delegating to `gsd-planner`. | blocking | Rework routes through `/gsd-plan-phase`; the orchestrator states *what* changed and lets the planner derive *how*. |
| Fixing code against a stale spec | Patching implementation to match new feedback while leaving CONTEXT.md saying the old thing. | blocking | Correct CONTEXT.md first, then replan from it. Already done for this rework (commit `6daec0c`). |
| Plan prose edits files not in `files_modified` | Three of the nine cross-AI findings existed because a plan's body described editing a file absent from its frontmatter, so wave conflict analysis could not see it. | advisory | When reviewing a plan, cross-check every file named in an `<action>` against the frontmatter list. |
| Unit fixtures that all share one shape | Every `config_sync` unit fixture was named `config.yaml`, so 988 unit tests could not catch a bug where the code ignored the caller's filename. Only the VM suite found it. | advisory | When a parameter carries a name or path, at least one test must use a value that differs from the convention. |
