    - src/pcswitcher/terminal.py — `is_interactive`, the shared interactivity predicate

  <behavior>
    - Interactive run: `review_items` calls `ui.pause()` before the prompt and `ui.resume()` after it, and returns exactly the selections the prompt reported.
    - Prompt raises: `ui.resume()` is still called (the `finally` branch), and the exception propagates.
    - Prompt returns `None` (user pressed Ctrl-C / aborted): every item comes back skipped-once and nothing is recorded permanently.
    - Non-interactive console: no prompt object is constructed at all, every item comes back skipped-once, and a warning naming the count of unresolved items is logged.
    - Grouping: given a mix of install, remove and change actions across two managers, the returned groups are keyed by (manager, action) with removals in their own group; no group mixes an install with a remove.
    - Group titles for removals name the concrete verb for the item class (remove / delete / disable), not the word "apply".
    - The blocking prompt call is dispatched through `asyncio.to_thread` — a test that runs `review_items` on a loop with a prompt stub that sleeps synchronously must not block the loop.
  </behavior>
  <action>
Run `uv add questionary` (or `uv add InquirerPy` if the checkpoint redirected), then create `src/pcswitcher/jobs/package_review.py`.

Module docstring: explain that this is the single interaction surface for every package job's diff (D-24), that it composes with the one persistent Live display by pausing it exactly as `TerminalUIConfirmer` does, and why removals get their own group (a bulk tick must never silently delete — D-07).

Public surface:
- A `ReviewGroup` frozen dataclass: `manager: str`, `action: DiffAction`-shaped string, `title: str`, `entries: Sequence[ReviewEntry]`. Define a minimal `ReviewEntry` protocol/dataclass here with the fields the prompt needs — `item_id: str`, `label: str`, `action_label: str`, `detail: str | None` — so this module does not yet depend on the item model that plan 02-03 introduces. Plan 02-05 adapts the real `ItemDiff` onto it.
- A `Decision` `StrEnum` with members `APPLY`, `SKIP_ONCE`, `SKIP_ALWAYS`.
- A `ReviewOutcome` frozen dataclass mapping `item_id -> Decision`, plus a `was_interactive: bool` flag so callers can tell a genuine skip-all from a user's choices.
- `async def review_items(groups: Sequence[ReviewGroup], *, console: Console, ui: PausableUI, logger: logging.Logger | None = None) -> ReviewOutcome`.

Behaviour of `review_items`:
- If `is_interactive(console)` is False: log one WARNING naming how many items went unresolved, print the groups to the console so the run still reports everything, and return a `ReviewOutcome` with every item mapped to `SKIP_ONCE` and `was_interactive=False`. Construct no prompt.
- Otherwise: call `ui.pause()`, and inside a `try`/`finally` that always calls `ui.resume()`, iterate the groups. For each group print a Rich `Panel` header carrying the group title and the concrete action verb, then run the checkbox prompt for that group's entries via `await asyncio.to_thread(...)`. Wrap any untrusted item text in `rich.text.Text` before putting it in a `Panel` so package names and stderr fragments are never parsed as Rich markup.
- Checkbox defaults: entries in an install-direction group default to checked; entries in a removal-direction group default to unchecked. Ticking is the affirmative act for a destructive item.
- A prompt returning `None` means the user aborted: map every remaining item to `SKIP_ONCE` and stop iterating further groups.
- Ticked entries map to `APPLY`; unticked entries map to `SKIP_ONCE`. `SKIP_ALWAYS` is not reachable from this primitive yet — plan 02-04 adds the second prompt that promotes a skip to permanent, and this module's return type already carries the member so that addition is additive.

Expose the module-level constant `PACKAGE_REVIEW_AUTOMATION_ENV = "PCSWITCHER_PACKAGE_REVIEW_AUTOMATION"` and read it in `review_items`: when the variable is set, take its value as a JSON mapping of `item_id -> decision` and return that outcome without prompting. Keep this undocumented — no mention in `--help`, no mention in `docs/configuration.md`, no entry in the config schema (D-26). Give it a one-line code comment saying it exists only so integration tests can drive the review, and that it is deliberately absent from user-facing surfaces.

Tests in `tests/unit/jobs/test_package_review.py` cover every bullet in the behavior block above, using a fake `PausableUI` recording pause/resume calls (same shape as `tests/unit/test_confirmer.py`), a non-TTY `Console(file=io.StringIO())` for the non-interactive case, and monkeypatched prompt stubs for the interactive cases. Do not drive a real terminal prompt in tests.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PyPI → developer machine | A third-party package enters the dependency tree and its code runs in-process on both machines. |
| item text → terminal renderer | Package names, versions and stderr fragments from package-manager output are rendered by Rich. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-SC | Tampering | `uv add questionary` | high | mitigate | Blocking, non-auto-approvable human checkpoint before the install; verifier checks the PyPI project links, release history and source repo against RESEARCH's `SUS` (`unknown-downloads`) verdict. |
| T-02-02 | Tampering | Rich `Panel` rendering of item text | medium | mitigate | Wrap untrusted item text in `rich.text.Text` before it reaches a `Panel`, so a package name or stderr fragment containing square brackets cannot be parsed as Rich markup and crash or reshape the display. |
| T-02-03 | Elevation of Privilege | `PCSWITCHER_PACKAGE_REVIEW_AUTOMATION` | medium | mitigate | The variable can only pre-answer a review; it can never widen what the review offers, and it is absent from `--help`, the config schema and user docs (D-26). Every decision it supplies is still confined to the items the diff already produced. |
</threat_model>
