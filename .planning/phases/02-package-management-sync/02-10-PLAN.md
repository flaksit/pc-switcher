<objective>
Wire the two remaining jobs into config, put the package jobs ahead of folder_sync, and hand folder_sync the paths those jobs own so the two stop fighting over the same directories.

Purpose: three jobs that work in isolation are still not a feature until the orchestrator runs them in the right order and the mirror stops overwriting what they install. D-17's ordering is decisive for flatpak — `~/.local/share/flatpak` must be created by `flatpak install` before folder_sync lands `~/.var/app` on top — and D-29 is what lets the user's hand-written filter lines go away.

Output: three enabled-able jobs, correct ordering, and automatic exclusions replacing hand-written filter rules.
</objective>

The existing VS Code exclusion is unconditional. These two are gated on their job being enabled, because unlike the VS Code DBs — which `vscode_state_sync` merges and therefore must be hidden from the mirror — a disabled snap or flatpak job means nobody is managing those paths, and excluding them anyway would strand that data unmirrored. Access it through a small helper rather than repeating the `(… or {}).get(name, False)` dance at both call sites.

Call both in `_build_rsync_cmd` in the GLOBAL-FIRST section, after the runtime, VS Code and decision-file exclusions and before the `merge` filter. Retiring the user's personal rules is a step in plan 02-12's human checkpoint, not something this task can do.

<threat_model>
## Trust Boundaries

| Boundary | Description |
| -------- | ----------- |
| job-supplied paths → rsync filter chain | An owning job's exported absolute paths become non-overridable exclusions in a destructive mirror. |
| config job order → execution order | The order of keys in a YAML block decides the order destructive operations run in. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
| --------- | -------- | --------- | -------- | ----------- | --------------- |
| T-02-25 | Denial of Service | exclusion gating | medium | mitigate | Snap and flatpak exclusions are gated on their job being enabled, so disabling a package job restores the mirror for those paths rather than leaving that data permanently unmirrored and unmanaged. |
| T-02-26 | Tampering | filter-rule authoring in `home.filter` | medium | mitigate | The replacement comment is on its own line — a trailing comment on an rsync filter rule becomes part of the pattern and silently disables it, which would re-expose paths the package jobs manage. |
| T-02-27 | Tampering | job execution order | medium | mitigate | A unit test pins the discovered job order for the shipped default config, so a future edit to `sync_jobs:` cannot silently move folder_sync ahead of the package jobs and let the mirror overwrite freshly-installed application defaults (D-17). |
| T-02-01 | Tampering | exported path interpolation into rsync filters | high | mitigate | `shlex.quote()` on every emitted `--filter=` value, matching the existing VS Code and runtime filter methods (ASVS V5). |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No package installs in this plan. |
</threat_model>
