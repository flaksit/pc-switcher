<objective>
The tracer: one path wired end to end through every layer this phase will touch — source manifest capture, target state query, item diff, the cross-manager plan/review/apply coordination, target convergence, per-item failure collection, progress and logging — for exactly one item class (apt package) in exactly one direction (missing on target → install).

Purpose: prove the architecture before ten layers are committed on top of it. Every later Phase 2 plan expands outward from this slice: more item classes, more diff directions, more managers, permanent decisions, snippets. If the manifest → plan → review → apply pipeline is wrong, it is wrong after one commit rather than after ten.

The coordination seam is part of that architecture, not a later refinement. D-15 wants three independent jobs and D-24 wants one batched review across all of them; those two only reconcile if a job's planning is separable from its applying. Wiring that seam here — while only one manager exists and it therefore looks like overkill — is precisely what stops `snap_sync` and `flatpak_sync` from being written against a shape that cannot satisfy D-24. Deferring it would mean discovering the conflict after three jobs are built on the wrong contract.

This is production code, not a prototype. It carries the same validation, dry-run contract, logging split and error handling as any finished job, and it survives into the shipped system unchanged in shape. What is missing is functionality (other item classes, other directions, other managers), never architecture.

Output: a working `apt_sync` job that installs missing packages on the target, the shared core the other two jobs will subclass, and the package-phase coordinator that guarantees one review precedes every manager's changes.
</objective>

`package_phase.py` — the coordinator.
- Module docstring: this module exists to satisfy two decisions that pull against each other. D-15 wants three independent jobs with their own config, enable flags, failure isolation and progress; D-24 wants one batched review across all of them before any change is applied. The coordinator resolves that by owning the one thing that must be global — the review — and leaving everything else with the jobs.

- `run()` does exactly three things in order. First, plan: `await job.plan()` for each job in turn, each wrapped so an exception is captured against that job rather than propagated — call `job.record_plan_failure(exc)` (add that small method to `PackageSyncJob` in task 1's module if it is not already there, or store the failure on the coordinator and hand it back through `accept_review`; prefer whichever keeps `execute()`'s "raise the stored failure" path in one place). Second, review: concatenate every plan's `ReviewGroup`s into one sequence ordered by manager (the supplied job order, which is config order and therefore D-17's order) and, within a manager, by a fixed action order, then `await review_items(...)` once. Third, distribute: for each job, build the subset of the `ReviewOutcome` whose item ids appear in that job's own plan and call `job.accept_review(plan, slice)`.
- The distribution step is the part worth a comment: the review is global but a decision belongs to exactly one manager, and item ids already carry their manager prefix, so the split is by membership in the job's own plan rather than by string parsing.
- Log one INFO line naming how many managers planned and how many items the review covered, and per-manager counts at FULL (ADR-010).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| package-manager output → command construction | Names and versions parsed from `apt-mark`/`dpkg-query` output are interpolated into subsequent shell commands. |
| source orchestrator → target root | Every converge command runs as root on the target via sudo over SSH (ADR-002, ADR-013). |
| target repositories → target system | The target downloads and installs packages from its own configured repositories (D-28). |

| T-02-01 | Tampering | `AptSyncJob.converge`, `capture_source_items`, `query_target_items` | high | mitigate | Every interpolated package name goes through `shlex.quote()` before reaching a command string, matching the existing `folder_sync`/`vscode_state_sync` convention (ASVS V5). |
