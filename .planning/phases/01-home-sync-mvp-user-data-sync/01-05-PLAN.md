Implement `FolderSyncJob.execute()` — the data plane. Build the rsync command (D-13 flags, D-11 filters, root-via-sudo per D-05), drive it as an async subprocess over its own SSH transport (ADR-002/ADR-005), stream progress and per-file events into the existing Rich TUI and the six-level logger (D-15/D-16), implement the dry-run preview (D-12), and record the post-sync divergence baseline (D-06/D-08).

Purpose: This makes a real `pc-switcher sync <target>` replicate the configured folders with full metadata, mirror semantics, and the machine-specific exclusions — satisfying success criteria 1-4 at the mechanism level. Integrity rests on rsync's built-in transfer verification (D-14), not a custom checksum pass.

Output: A complete, working FolderSyncJob with unit coverage of command construction, streaming, dry-run, and marker recording, plus a contract-suite entry.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user/config → rsync command line | Folder paths and exclude patterns become rsync arguments. |
| source → target (SSH) | rsync opens its own SSH transport to the target (ADR-002); orchestration uses asyncssh separately. |
| user → root (sudo) | rsync is elevated to root on both ends via sudo. |
| target filesystem ← mirror delete | `--delete` removes target files not present on the source. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Tampering (command/arg injection) | folder paths + exclude patterns mapped onto the rsync command line and --rsync-path | high | mitigate | Every config-derived value (path, pattern, hostname) is `shlex.quote`d; schema (plan 02) constrains them to strings. Unit tests cover the constructed command. |
| T-05-02 | Elevation of privilege | rsync-as-root via passwordless sudo (D-05) | high | mitigate | Sudoers entry scoped to the rsync binary path only (NOT ALL / no wildcards) — documented in user_setup and the ADR-013 record; root SSH login stays disabled; rsync reached via --rsync-path='sudo rsync' over the normal-user SSH connection. |
| T-05-03 | Tampering / Data loss | `--delete` mirror semantics | high | mitigate | Gated behind the divergence guard (plan 04) which runs in validate() before any transfer; btrfs pre/post snapshots are the backstop. `--delete-excluded` is never passed so excluded files on the target are not deleted. |
| T-05-04 | Information disclosure | secret leakage of .ssh/id_*, .config/tailscale | high | mitigate | These are shipped as default excludes (plan 02) and mapped to `--filter` rules here; `--delete-excluded` is never used (they survive on the target); plan 06 asserts they never appear on the target. |
| T-05-05 | Spoofing / MITM | SSH transport trust | medium | mitigate | rsync uses system ssh honoring ~/.ssh/config and known_hosts (ADR-002); no custom transport. |
| T-05-SC | Tampering | npm/pip/cargo installs | low | accept | No Python packages added; rsync + acl are standard Ubuntu system packages validated in plan 04's validate() (RESEARCH Package Legitimacy Audit). |
</threat_model>
