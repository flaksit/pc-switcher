<objective>
Stop the live-progress flooding by funnelling all interactive terminal output through the single rich.live.Live. Replace the TUI-floor stderr StreamHandler with a logging.Handler that routes records into TerminalUI's existing Recent Logs panel (via the event loop, so every terminal write goes through one Live.update path). Keep the plain stderr StreamHandler only for non-interactive / non-TTY runs.

Purpose: Root cause confirmed by live raw-byte reproduction (see .planning/debug/tui-live-progress-flooding.md): setup_logging installs logging.StreamHandler(sys.stderr) which writes formatted records straight to the terminal on the QueueListener thread, uncoordinated with Live's cursor bookkeeping. Any INFO+ log while Live is active (routine orchestrator phase logs, dummy_success's periodic logs) desyncs Live's "lines last rendered" counter, so its next redraw erases the wrong region and leaves stale duplicate frames — 761 duplicate panel headers and 326 duplicate 0% frames in one run. Closes UAT gap 4.

Output: A UI-routed log handler, a TTY-aware setup_logging, an orchestrator that wires them, and tests.
</objective>

    Add to logger.py a small Protocol (e.g. LogPanelSink) with a single method add_log_message(message: str) -> None, so the handler does not import TerminalUI directly (ui.py already satisfies this shape).

    Add a logging.Handler subclass (e.g. UILogHandler) whose emit(record) formats the record to a compact PLAIN-TEXT single line (no ANSI escape codes — the Recent Logs Panel renders strings, and raw ANSI would show literally, while Rich markup would risk markup-injection from arbitrary messages) containing: HH:MM:SS time, level name, optional [job] and (host) drawn from the record's extra fields (omit when absent), and the message. Capture the loop with asyncio.get_running_loop() at handler construction (the handler is only ever constructed from within the running orchestrator loop). Guard emit() so a closed/stopped loop (late records during shutdown) is swallowed via self.handleError(record) rather than raising.

Decide: use_ui = ui is not None and the console indicates a real terminal (console.is_terminal). When use_ui, the second handler is UILogHandler(ui) at level log_config.tui; otherwise keep the existing logging.StreamHandler(sys.stderr) with RichFormatter at level log_config.tui. The FileHandler (JSON, level log_config.file) is unchanged, and both handlers still go through the same QueueListener with respect_handler_level=True. Keep the signature backward-compatible (ui/console default None → stderr StreamHandler, so existing callers/tests are unaffected).

| log message text → Recent Logs panel | arbitrary message content is rendered in a Rich Panel |
