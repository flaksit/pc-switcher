<objective>
Close the Phase 1 BLOCKER and the two data-loss-linchpin warnings in the target-divergence guard.

- CR-01 (BLOCKER, D-07/T6): the default `/home` config (empty subvolume prefix) self-triggers a false divergence on every second sync because pc-switcher's own post-sync writes (`sync-history.json` and the remote lock file, both under `~/.local/share/pc-switcher/` inside `@home`) bump the target's `@home` generation after the baseline is captured. On the next sync, `btrfs find-new` surfaces those tool-state files and, with `prefix==""`, any changed file counts as divergence — blocking the sync. Fix by excluding the pc-switcher state directory from the divergence scope.
- CR-02: the guard fails OPEN (proceeds with destructive mirror+`--delete`) when `findmnt`/`find-new` fails even though a baseline exists. Make it fail CLOSED in that case; keep fail-open only for the no-baseline/never-synced case (RESEARCH Open Q3).
- WR-02: a post-transfer baseline-capture exception aborts an otherwise-successful sync AND leaves the next run with no baseline (guard silently skipped). Make baseline recording non-fatal and record a sentinel so the next run is guarded.

Purpose: make the shipped default configuration usable for repeated syncs and restore the divergence guard to a real data-loss safety control rather than one that false-positives or fails open.

Output: corrected divergence state machine in `folder_sync.py`, a sentinel constant in `sync_history.py`, offline unit coverage, and a VM-executable integration scenario.
</objective>
