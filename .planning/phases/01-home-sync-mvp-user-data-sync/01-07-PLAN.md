<objective>
Close the Phase 1 BLOCKER and the two data-loss-linchpin warnings in the target-divergence guard.

- CR-01 (BLOCKER, D-07/T6): the default `/home` config (empty subvolume prefix) self-triggers a false divergence on every second sync because pc-switcher's own writes bump the target's `@home` generation after the baseline is captured. Two pc-switcher-owned target write sites land inside `@home`: (1) post-sync state under `~/.local/share/pc-switcher/` (`sync-history.json` and the remote lock file), and (2) the Phase-8 config-sync write of `~/.config/pc-switcher/config.yaml` (orchestrator runs config sync BEFORE job execution; see `config_sync._copy_config_to_target`, gated on configs differing / target missing config). On the next sync, `btrfs find-new` surfaces those tool-state files and, with `prefix==""`, any changed file counts as divergence — blocking the sync. Fix by excluding BOTH pc-switcher state directories from the divergence scope — but only when the folder prefix is empty (subvolume root). The filter MUST NOT fire for a non-empty synced prefix (a user syncing a folder that happens to contain a `.local/share/pc-switcher/`-looking subpath must still get real changes detected — Codex HIGH #2).
- CR-02: the guard fails OPEN (proceeds with destructive mirror+`--delete`) when `findmnt`/`find-new` fails even though a baseline exists. Make it fail CLOSED in that case; keep fail-open only for the no-baseline/never-synced case (RESEARCH Open Q3).
- WR-02: a post-transfer baseline-capture exception aborts an otherwise-successful sync AND leaves the next run with no baseline (guard silently skipped). Make baseline recording non-fatal and record a sentinel so the next run is guarded.

Purpose: make the shipped default configuration usable for repeated syncs and restore the divergence guard to a real data-loss safety control rather than one that false-positives or fails open.

Output: corrected divergence state machine in `folder_sync.py`, a sentinel constant in `sync_history.py`, offline unit coverage, and a VM-executable integration scenario.
</objective>
