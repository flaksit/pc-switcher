<objective>
Close the remaining folder_sync observability and time-of-check/time-of-use findings.

- WR-01: `bytes_transferred` is always 0 in the per-folder INFO summary because `_PROGRESS2_RE` never captures the leading byte figure and `bytes_xfr` is never reassigned — a misleading audit trail.
- IN-03: `_stream_rsync` recognizes per-file `%i` lines only for `>`/`<`/`*`/`.`; rsync also emits `c` (created dirs/symlinks/devices) and `h` (hard links), which are silently dropped from FULL logging.
- WR-03 (TOCTOU): the divergence check runs at orchestrator Phase 4 but the destructive `rsync --delete` runs at Phase 9 — a multi-phase window in which target writes escape the guard. Re-check divergence immediately before the destructive transfer.

Ordering caveat (Codex HIGH, source-confirmed): the orchestrator runs Phase 8 "config sync" (`config_sync._copy_config_to_target` → writes `~/.config/pc-switcher/config.yaml` when configs differ / target missing config) BEFORE Phase 9 job execution. For the default `/home` (empty subvolume prefix), that config.yaml write is inside `@home` and would be surfaced by the execute-time `find-new` re-check. This plan's re-check therefore depends on 01-07 having broadened AND scoped the tool-state filter to also exclude `~/.config/pc-switcher/` for the empty-prefix case; otherwise the re-check reintroduces a false divergence right after pc-switcher itself updated the target config. This plan adds explicit unit coverage for that config.yaml case.

Purpose: a truthful audit trail (REQ-terminal-ux logging) and a tighter data-loss guard (REQ-manual-sync-workflow / REQ-sync-scope-user-data) that re-validates the target right before it mirrors with deletes — without re-flagging pc-switcher's own Phase-7/8 target writes as divergence.

Output: corrected progress parsing and itemize handling in `_stream_rsync`, and a pre-transfer divergence re-check in `execute()`.
</objective>
